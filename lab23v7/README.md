Аналіз початкового коду виявив порушення принципів ISP та DIP.

Початковий інтерфейс IDocumentProcessor містив 5 методів, але класи-клієнти використовували лише частину з них. Це порушувало ISP. Клас ReportGenerator створював залежності всередині себе, що порушувало DIP.

Після рефакторингу великий інтерфейс розділено на чотири спеціалізовані: IFileReader, IFileWriter, IFilePrinter, IFileConverter. Кожен клас реалізує лише необхідні йому інтерфейси.

Реалізовано Dependency Injection через конструктор. Клас ReportGenerator отримує залежності ззовні, що дозволяє легко міняти реалізації та тестувати код.

Програма демонструє роботу з трьома типами документів: Word, PDF та Excel. Кожен тип має власні реалізації інтерфейсів.

Висновок: Застосування ISP та DIP покращило гнучкість, тестованість та підтримуваність коду.
